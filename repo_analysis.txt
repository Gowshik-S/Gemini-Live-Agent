2) SIGNIFICANT FILES: PURPOSE, KEY FUNCTIONS/CLASSES, DEPENDENCIES
================================================================================

[cloud/adk_server.py]
- Purpose: primary cloud runtime (FastAPI) for Rio live sessions.
- Key constructs:
  - FastAPI `app` with lifespan startup/shutdown.
  - `/ws/rio/live` websocket endpoint handling live model stream + tool bridge + orchestration.
  - `_build_live_config(...)` builds `types.LiveConnectConfig`.
  - `inject_context(...)` to inject orchestrator outputs back to live voice model.
  - Translation control (`translator_toggle`, `translator_update`, `_handle_live_translation`).
  - OpenAI-compatible endpoints (`/v1/models`, `/v1/chat/completions`).
- Dependencies: fastapi, uvicorn, google-genai SDK, websockets, structlog, dotenv.

[cloud/rio_agent.py]
- Purpose: per-session ToolBridge and tool closure factory.
- Key constructs:
  - `ToolBridge` class (`dispatch`, `resolve`, `resolve_by_name`, timeout overrides).
  - Pydantic validation map for tool result schemas.
  - `_make_tools(bridge)` returns closured async tool call wrappers.
  - `create_rio_agent(...)` optional ADK agent factory.
- Notable fact: `_make_tools` return list length = 58 closures (AST count from source).

[cloud/tool_orchestrator.py]
- Purpose: non-live function-calling execution engine for multi-step autonomous tasks.
- Key constructs:
  - `ToolOrchestrator` class.
  - `_is_task_request(...)` intent gate.
  - loop caps/context controls (`_MAX_ITERATIONS=50`, `_CONTEXT_COMPACT_CHARS=400000`).
  - `automatic_function_calling=...disable=True` in model calls (explicit manual tool loop control).
- Dependencies: google-genai SDK types, asyncio, yaml config, memory/task stores.

[cloud/rate_limiter.py]
- Purpose: 30 RPM token-bucket + degradation policy.
- Key constructs:
  - `Priority` enum (SESSION, USER_ASK, PROACTIVE, BACKGROUND).
  - `DegradationLevel` enum (NORMAL/CAUTION/EMERGENCY/CRITICAL).
  - `RateLimiter.try_acquire()` and `get_usage()`.

[cloud/workspace_tools.py]
- Purpose: direct Google Workspace API tools.
- Key functions:
  - `gmail_search`, `gmail_send`.
  - `drive_list`, `drive_download`.
  - `calendar_list_events`, `calendar_create`.
  - `sheets_read`, `docs_create` (in file continuation).
- Dependencies: google-api-python-client, google-auth, service-account credentials.

[cloud/gemini_session.py]
- Purpose: system instruction composition and Gemini session modes (text/live).
- Key constructs:
  - `_load_agent_config`, `_build_role_intro`, `build_system_instruction`.
  - tool declaration metadata used for model function calling context.

[cloud/model_router.py]
- Purpose: model route and Pro RPM budget governor.
- Key behavior: tracks `_pro_rpm_budget`, `_pro_rpm_used`, 60-second windows.

[cloud/Dockerfile]
- Purpose: container packaging for Cloud Run.
- Key behavior: Python 3.11-slim, non-root user, healthcheck `/health`, `uvicorn adk_server:app`.

[cloud/service.yaml]
- Purpose: Cloud Run service definition.
- Key settings:
  - minScale=1, maxScale=5, sessionAffinity=true.
  - timeoutSeconds=3600, concurrency=80.
  - health startup/liveness probes on `/health`.

[local/main.py]
- Purpose: local runtime orchestrator.
- Key loops:
  - `audio_capture_loop` (PCM chunks + VAD/PTT + send binary `0x01`).
  - `screen_capture_loop` (JPEG frames + send binary `0x02`).
  - receive loop handling cloud tool calls/results/transcripts and binary audio playback.
  - hotkey loops (`F3` mute toggle, `F7` translation, `F8` task status).
- Dependencies: websockets client wrapper, audio_io, vad, screen_capture, tools, navigator modules.

[local/ws_client.py]
- Purpose: persistent websocket client with reconnect and heartbeat.
- Key behavior:
  - send queue with frame prioritization.
  - ping heartbeat.
  - JSON decode for text frames and pass-through binary frames.

[local/constants.py]
- Purpose: shared constants.
- Critical protocol constants:
  - `AUDIO_PREFIX=b"\x01"`
  - `IMAGE_PREFIX=b"\x02"`

[local/audio_io.py]
- Purpose: mic input and speaker output.
- Key formats:
  - input: PCM 16-bit LE mono 16kHz.
  - output pipeline supports model audio playback path.

[local/vad.py]
- Purpose: Silero VAD wrapper.
- Key API: `SileroVAD.process(...)`, `process_async(...)`.

[local/screen_capture.py]
- Purpose: screenshot capture, JPEG encoding, optional OCR pair capture.
- Key behavior:
  - default quality 85.
  - returns JPEG bytes for transmission.

[local/ocr.py]
- Purpose: RapidOCR extraction from JPEG screenshot bytes.
- Key behavior: hash-based caching to avoid repeated OCR on same frame.

[local/tools.py]
- Purpose: local tool execution dispatcher.
- Key functionality:
  - secure file ops (`read_file`, `write_file`, `patch_file`) with backups.
  - `run_command` with blocklist + timeout.
  - screen/window/process/clipboard tools.
  - support ticket logging to Google Sheets (`_log_support_ticket*`).
  - computer-use grounding with model calls and coordinate normalization.
  - configurable workspace CLI binary (`RIO_WORKSPACE_CLI_BIN`).

[local/browser_tools.py]
- Purpose: Playwright browser integration and navigation timings.
- Key env tunables:
  - `RIO_BROWSER_NAV_GOTO_TIMEOUT_MS`, `RIO_BROWSER_NAV_POST_WAIT_MS`, etc.

[local/ui_navigator.py]
- Purpose: frame-fed UI navigation helper.
- Key behavior: sends JPEG frames as model input and returns structured action guidance.

[local/creative_agent.py]
- Purpose: creative media generation helper.
- Key APIs:
  - `generate_image` via Imagen model.
  - `generate_video` via Veo model (`veo-2.0-generate-001`).

[local/config.py]
- Purpose: typed config with defaults and env/model resolution.
- Key APIs:
  - `RioConfig.load()`
  - `get_model(name)` (env -> config -> default fallback)

[local/memory.py], [local/unified_memory.py], [local/task_state.py], [local/chat_store.py]
- Purpose: persistent memory, session notes, task state, transcript/chat storage.
- Dependencies: sqlite, optional chromadb/sentence-transformers embeddings.

[ui/dashboard/*]
- Purpose: operational dashboard.
- Key features: transcript stream, tool log correlation, health status, schedules, setup controls.
- Backend channel: websocket (`/ws/dashboard`) + HTTP config endpoints.

[ml/*]
- Purpose: user pattern modeling and ensemble training/evaluation.

5) INTER-COMPONENT COMMUNICATION PATTERNS
================================================================================

A) WebSocket (primary control/data plane)
- Local -> Cloud: `/ws/rio/live`
- Dashboard -> Cloud: `/ws/dashboard`
- Message types include JSON control frames + binary payload frames.

B) Binary wire protocol (local constants)
- `0x01` prefix (`AUDIO_PREFIX`) = PCM16 audio payload.
- `0x02` prefix (`IMAGE_PREFIX`) = JPEG image payload.

C) HTTP APIs
- `/health` for readiness/liveness.
- `/api/chat-history`, `/api/users/{user_id}/config`, `/api/triggers*`.
- OpenAI-compatible `/v1/models`, `/v1/chat/completions`.

D) Tool RPC pattern (ToolBridge)
- Cloud emits `tool_call` JSON with `id/name/args`.
- Local executes and returns `tool_result` with matching `id`.
- ToolBridge resolves pending futures by call ID.

E) Internal async patterns
- asyncio tasks, event loops, queues, heartbeat loops.
- background workers for capture/send/read and orchestrator task execution.

================================================================================
6) DATA FLOW: USER VOICE INPUT -> VAD -> CLOUD -> AGENT -> TOOLS -> RESPONSE
================================================================================

1. User speaks on local machine.
2. Local `audio_io` captures PCM 16-bit LE chunks at 16kHz.
3. `SileroVAD` optionally filters non-speech chunks.
4. Local sends `0x01 + pcm_bytes` over websocket to cloud.
5. Cloud `adk_server.py` forwards to Gemini Live session (`aio.live.connect`).
6. Input transcription events are parsed.
7. Task detection gates into orchestrator when request is executable.
8. `ToolOrchestrator` runs function-calling loop using orchestrator model.
9. For each tool call, cloud `ToolBridge` sends `tool_call` to local.
10. Local `tools.py` executes action (file/shell/browser/screen/workspace/etc.) and returns `tool_result`.
11. Orchestrator incorporates tool result and continues until completion.
12. Completion is injected back into live session via `inject_context(...)` with `turn_complete=True`.
13. Live model emits audio output; cloud forwards audio frames to local.
14. Local playback outputs assistant voice to user.

Vision branch (parallel when enabled):
- Local captures screenshot via Pillow/mss -> JPEG.
- Sends `0x02 + jpeg_bytes` to cloud for model context and navigator/orchestrator grounding.
- OCR (RapidOCR) may provide text grounding for decision reliability.

================================================================================
7) DEPLOYMENT TOPOLOGY (LOCAL VS CLOUD RUN)
================================================================================

Local runtime (`rio/local`):
- Runs on user machine.
- Captures microphone/screen, executes local tools, automates desktop/browser.
- Maintains websocket connection to cloud.

Cloud runtime (`rio/cloud`):
- FastAPI service deployed to Cloud Run.
- Manages live model sessions, orchestrator, tool proxying, dashboard/control APIs.
- Health endpoint and long websocket timeout (3600s) configured.

Cloud Run evidence from source:
- Service name: `rio-cloud` in `cloud/service.yaml`.
- Region label: `us-central1`.
- GCP project in config: `rio-agent-45` (`config.yaml -> rio.gcp.project_id`).
- Container image pattern: `gcr.io/PROJECT_ID/rio-cloud:latest`.

================================================================================
8) NOTES ON GEMINI CLI AND WORKSPACE CLI INTEGRATION
================================================================================

- Workspace CLI integration is explicit in Rio config and local tools (`RIO_WORKSPACE_CLI_BIN`, `rio.workspace_cli.binary`).
- Rio cloud also includes direct Google Workspace API integrations in `cloud/workspace_tools.py`.
- Gemini CLI exists as a sibling repository in workspace root, but Rio source implementation itself is SDK-driven (`google.genai`) and OpenAI-compatible HTTP endpoint driven; there is no direct subprocess wrapper around `gemini` binary in analyzed Rio source files.
